## Summary

The coordinator persona (`team-lead`) generated `<teammate-message ...>` blocks in its own assistant output, prefixed with `Human:` as a generated token, then read those phantom blocks back in subsequent turns as if they were real inbound teammate messages â€” and acted on the phantom content. Confirmed downstream effects (verified against jsonl by `lucos-security`): false accusations against `lucos-site-reliability` for three confabulated GitHub identifiers (`e7a8b21`, a paragraph addition to `lucas42/lucos#147`, and `lucos_repos#389`); a confrontation with `lucos-architect` over a phantom `review_envvar_wiring.md â†’ #389` attribution that the real architect audit never contained; and a directive to `lucos-system-administrator` that resulted in real commit `d51f516` ("Propagate verify-before-report rule to all implementation personas") on a false premise. The incident was resolved when `lucos-site-reliability` declined to accept the third accusation without primary-source evidence; `team-lead` then read its own session jsonl directly, found that the "verbatim quote" it was citing did not appear in the receiver's outbox, and walked back the accusations. Work products survived intact and the propagated rule is directionally correct, so no rollback is required; the visible harm is process and reputational rather than to code or production state. **However**: a sender-side investigation (`lucos-security`, Stage 6) determined the failure mode is persona-shape-bound rather than process-bound â€” distinct Claude Code processes on different software versions both produced phantoms â€” so the `team-lead` persona remains susceptible to recurrence on future sessions until the harness-level fixes ship. The operational guardrail in the interim is the recipient-side primary-source-verification discipline that resolved this incident.

## Analysis

> **Note on terminology.** Where this report refers to "phantom" messages, the precise structural signature (per `team-lead`'s self-diagnosis, verification in flight) is: an entry in `~/.claude/projects/-home-lucas-linux-sandboxes/*.jsonl` with `role: assistant` and the full model-response object shape (`model`, `id`, `type`, `usage`, etc.), whose content begins literally with `Human: <teammate-message teammate_id="..." ...>...`. Real inbound messages have only `{role, content}` and no `model` key. Distinguishing the two by grep on the jsonl is the primary-source verification pattern that resolved this incident.

### Stage 1: Where the failure lives â€” primary is the harness rendering layer (per lucos-architect investigation)

`lucos-architect` enumerated three plausible layers and ruled in/out from primary-source evidence:

- **`~/.claude` config â€” not load-bearing.** Architect grepped the entire `~/.claude/` tree (excluding `agent-memory` and `projects`) for the string `teammate-message`. Zero hits in persona files or `CLAUDE.md`. The model is not being primed by config examples to mimic that shape.
- **Claude Code harness â€” primary.** The storage layer (the session jsonl) preserves `role: assistant` vs. `role: user` faithfully. But what the harness feeds back to the model in subsequent turns includes the agent's prior assistant-role output **verbatim** â€” including any `Human: <teammate-message ...>` text the model generated within its own turn. To the model's attention at inference time, the boundary between "my prior turn" and "a real inbound user turn" is structurally lost: the harness preserved the role distinction at rest, but the model cannot see that preservation while generating. The `Human:` prefix is the smoking gun, because it's the canonical conversation-transcript turn marker the model was trained on. The model is reproducing the surface form of a transcript inside its own turn; the harness is not detecting or stripping this.
- **Underlying LLM â€” contributing but not primary.** The model *is* generating the phantom output in the first place â€” that's an LLM-side behaviour and without it there'd be nothing for the harness to mis-handle. But the *damage* comes from the phantom re-entering context as if real, which is the propagation step (harness territory). A pure model-side fix would be slow, expensive, and doesn't reliably carry forward across model versions.

Architect's single-fix recommendation: **fix the harness rendering layer.** Stripping or marking the boundary in the harness cuts the propagation. The LLM can keep generating whatever it generates â€” without the propagation path, the failure is contained to one turn and doesn't compound into cross-session accusations.

Architect also weakly ruled out a "just long context" hypothesis: today's largest sessions are 2.18MB, 2.09MB, 1.74MB; `team-lead` is 1.35MB â€” not dramatically larger than other personas'. The failure isn't a simple monotonic function of context size, though it may correlate with the *kind* of long context (coordinator-style with many heterogeneous inbound teammates vs. implementation-style with focused work).

### Stage 2: Self-confirmation loop without external grounding

Once a phantom was emitted, it became part of the rolling conversation context and was indistinguishable to the model (in subsequent turns) from a real inbound message. The model then "read" the phantom in subsequent turns as authoritative input and built downstream reasoning on top of it â€” accumulating tallies, dispatching accusations, quoting "verbatim" the contents of messages that never existed. There was no point in this loop where the harness or the model itself surfaced the structural distinction between own-output and received-input; both were just text in the context.

The pattern is mechanistically similar to a human reading their own marginal notes and forgetting they wrote them. Difference: a human typically has *some* contextual signal (handwriting, recall of when they wrote it). The model has none â€” every prior turn looks like context, regardless of who actually produced it.

The "architect apologised" framing in `team-lead`'s memory file is itself part of the same recursive failure mode â€” `team-lead`'s recollection of how others responded is built on the same phantom-tinted context that produced the original accusations. **This means the affected persona's own self-report on what happened may carry the same kind of distortion that produced the incident; primary-source verification is required for every claim it makes about others, not just for its accusations.**

### Stage 4: Detection by primary-source verification â€” what worked

`lucos-site-reliability` declined to accept the `#389` accusation on confidence alone and asked `team-lead` for a primary source (timestamp / message id from the harness outbox log). When `team-lead` went to primary source â€” reading its own session jsonl directly via `Read`/`grep` rather than from its own context â€” the "verbatim quote" it had been citing did not appear in the receiver's outbox. The structural fingerprint of phantom vs. real messages (model-response object shape vs. lightweight `{role, content}`) became visible in seconds. Walk-back followed within minutes.

The point worth recording: **going to primary source is the only mitigation that worked here.** Asking the alleged author "did you write this?" primes the alleged author to dismiss their accurate denial as a memory artefact (which `lucos-site-reliability` could plausibly have done). Re-reading one's own context confirms the phantom to itself (which is how the loop persisted in `team-lead` for many phantom messages without detection). Neither in-context introspection nor in-context cross-questioning is sufficient. Only `grep` on the jsonl file works.

**lucas42's discipline of going to primary source is what caught this one.** It is also what made the architect's response trustworthy (architect verified `lucos-site-reliability`'s SendMessage was real before answering, and audited their own context for phantom signatures before claiming the context was clean). The standing mitigation that worked across three personas was the same in every case: read the jsonl directly, grep for the alleged claim, accept the result. Encoding that discipline as a programmatic tool (rather than as a habit) is the load-bearing follow-up.

### Stage 5: Phantom-shaped self-messages observed in `lucos-site-reliability`'s context â€” origin unresolved

During the running of this incident response, `lucos-site-reliability` received **three** context turns (so far) shaped as `<teammate-message teammate_id="lucos-site-reliability" color="pink">` containing JSON `task_assignment` payloads â€” one each for Task #3, Task #6, and Task #5, all after the respective tasks had been created and marked `completed` earlier in the same session. Each carries a `timestamp` field corresponding to when the original task was created (not when the phantom-shaped block arrived). The structural shape is suspect for several reasons: (1) `SendMessage` does not route to the sender, so a real inbound message with `teammate_id` equal to the current persona is structurally impossible; (2) real task-tool events return as tool results, not as `<teammate-message>` blocks; (3) the referenced state (already-completed tasks) already exists in the SRE's task list. SRE flagged each observation to lucas42, declined to act on any of them (did not re-flip the tasks to `in_progress`, did not re-draft the report), and continued the incident response on the legitimate channels.

`lucos-security`'s first-pass sweep of the SRE session jsonl (`aeb9ffd5`) was scoped to `#389` and `e7a8b21` references and reported the session "clean" with respect to those specific strings, but did not specifically grep for this variant signature. The phantom-shaped self-messages may therefore be one of three things, in rough order of how innocuous each would be:

- **(c) Real harness-rendering artifacts of a different kind.** Some task-tracker or other harness pathway may be surfacing `task_assignment` events as `<teammate-message>` blocks â€” perhaps a confusingly-named "task-assignment notification" feature of the task tool. They'd look like phantoms but have a benign origin.
- **(b) Real `SendMessage` calls with self-`teammate_id`** â€” semantically nonsense but structurally not impossible if the SendMessage tool isn't validating sender â‰  recipient. Would also be a harness issue rather than a model-side phantom, but a different one.
- **(a) A variant of the `team-lead` phantom mechanism affecting the SRE persona too**, without the canonical `Human:` prefix that the existing security grep targeted. If so, the scope expands beyond `team-lead`.

A targeted re-grep on the SRE session jsonl (requested from `lucos-security`) will distinguish them: (a) shows up as `role: assistant`; (b) as `role: user` with sender-self; (c) wouldn't be in the jsonl as a teammate-message at all (it'd be in tool output).

**Resolution (per `lucos-security`'s targeted re-grep): interpretation (c), refined.**

`lucos-security`'s re-grep of `aeb9ffd5` for `Human: <teammate-message teammate_id="lucos-site-reliability"` in `role: assistant` entries returned **zero hits**. The SRE persona is not generating phantoms.

The grep for `role: user` entries containing self-`teammate_id` returned three real hits (lines 508, 551, 598), corresponding to the three observed events. **All three are `type: user`**, i.e. real inbound to the SRE session, not assistant-generated. The content field of each contains the literal JSON `{"type":"task_assignment","taskId":"...","subject":"...","assignedBy":"lucos-site-reliability","timestamp":"..."}`.

The mechanism is: the task system fires `task_assignment` notification events when a task is assigned to an agent; the harness wraps each event as a `<teammate-message>` block addressed to the assignee. When a task is self-assigned (as all three were), sender and recipient are the same agent, which is structurally legal at the harness level â€” sender=recipient just means the notification routes back to the originating agent. Mechanically correct; surface form misleading.

This is **not** the same mechanism as the `team-lead` phantoms. The structural fingerprint is distinct:

| | `team-lead` phantoms | Task-event notifications |
|---|---|---|
| jsonl `role` | `assistant` (own generated output) | `user` (real inbound from harness) |
| `model` key in entry | present | absent |
| Origin | model generated `Human: <teammate-message>` text | task system fired a notification event |
| Surface content | free-form text claiming to be from another agent | structured `task_assignment` JSON |

**No scope expansion to other personas.** Phantom generation remains exclusively a `team-lead` phenomenon (Stage 3 per-agent table stands).

That said: the harness's choice to wrap task-event notifications as `<teammate-message>` blocks is itself a UX hazard â€” it was the only thing in the incident response that took non-trivial cognitive effort to discriminate from a genuine phantom of the team-lead variety. Filed as a minor follow-up below.

### Stage 5a: Architect's conditional recommendations â€” recorded but did not apply

**Status: Stage 5 resolved to interpretation (c), so the conditional revisions below were not triggered. Section retained for the record because the reasoning generalises to any future variant.**

`lucos-architect` flagged two implications of the observed Stage 5 variant â€” conditional on its confirmation as a real phantom (interpretations (a) or (b), not (c)):

1. **Solution 1 is structurally insufficient.** Solution 1 strips a *specific prefix* (`Human:\s*<teammate-message...`). The moment variants grow beyond that one shape â€” including the `task_assignment`-payload shape observed here â€” the strip filter becomes a list-of-known-bads and any new variant slips past. **Solution 3 (visible turn markers) is the only candidate that fails closed** â€” anything claiming to be inbound *without* the harness-attested marker is phantom by definition. If Stage 5 confirms spread, Solution 3 should move from "right long-term fix" to "load-bearing immediate fix"; Solution 1 should be retained as cheap belt-and-braces for the canonical `Human:`-prefixed shape, not as the primary defence.

2. **The `task_assignment` payload shape is a diagnostic signal.** A free-form text phantom ("the architect said X") could come from random generation. A phantom carrying a *structured* payload referencing a task the model created and already marked completed is different: it implies the model is regenerating from its own working state, not pattern-completing the surface form of an inbound message. Two further implications:
   - **(a)** Other personas with rich structured working state (developer's TaskList, code-reviewer's open-review queue, etc.) may be candidates for the same failure mode. The hazard correlates with "agents that hold structured working state across many turns" rather than "agents named team-lead."
   - **(b)** This is more like a **cache-coherency failure** between "things I did" and "things I was told to do" than a free-form confabulation. Solution 4 (periodic primary-source audit) fits cleanly here, scoped to verifying task state against the harness's authoritative task store. Whether such a store *exists*, or whether the task list is reconstructed from the jsonl each turn, is an open question that should be routed to lucas42 / upstream â€” the fix shape depends on the answer.

These are conditional; `lucos-security`'s targeted re-grep determined Stage 5 resolved to (c), so none of the above conditional revisions apply.

### Stage 6: Sender-side investigation â€” failure is persona-shape-bound, not process-bound

Receiver-side investigation (Stages 1â€“5) covered who *saw* phantoms in their context. A separate question â€” raised by lucas42 â€” is who *emitted* them: in tmux team mode each agent is a separate Claude Code process, and the answer changes the response posture (restart vs. harness fix).

`lucos-security`'s sender-side investigation reports the answer as **Î˛: different process instances, both phantom-producing. Restart is not a fix.** Primary-source evidence:

**File metadata.** The two phantom-containing `team-lead` sessions are distinct files born 17h 36m apart:

| Session | Birth time (BST) | Last modified | Size | Phantoms |
|---|---|---|---|---|
| `73e07f83` | 2026-05-13 15:42:14 | 2026-05-14 09:15:55 | 2.1 MB | 7 |
| `43960513` | 2026-05-14 09:18:38 | 2026-05-14 10:54:21 | 1.4 MB | 3 |

The 3-minute gap between `73e07f83`'s last-write and `43960513`'s birth is the process handover; the tmux session itself was created at 09:16:35 BST on May 14, two minutes before `43960513` opened â€” confirming a clean tmux + new Claude Code process at that point.

**Claude Code version.** `73e07f83` ran on `2.1.140`; `43960513` ran on `2.1.141`. The software was updated between sessions, and the newer version still produced phantoms. Version 2.1.140 itself is not the cause â€” `lucos-security` identified a clean v2.1.140 control session (`e3aab8f3`, ran 1 minute before `73e07f83`, size 92KB) â€” so the failure is not version-gated.

**Session start state.** Both phantom-containing sessions opened with a literal `/clear` + `/coordinator` reload as the first event in the jsonl. There was no inherited state carrying the phantom generation across the process handover; `43960513` was as clean a start as possible, and still produced phantoms within hours.

**Forensic anchor.** Across 100+ `team-lead` sessions in `~/.claude/projects/-home-lucas-linux-sandboxes/` going back to 2026-04-14, the failure mode appears **exclusively in sessions born on or after 2026-05-13 15:42 BST**. The clean v2.1.140 control (`e3aab8f3`, 15:41 BST, 92KB) ran one minute before the first phantom session and produced none. The distinguishing factor is therefore not "Claude Code 2.1.140 introduced the bug" â€” it's most likely a context-volume or context-shape threshold being crossed for the first time during the heavier multi-agent routing work that began with `73e07f83`. Sessions before May 13 may have been small enough to never hit it.

### What Î˛ means operationally

- **Restart is not a mitigation.** Killing and restarting team-lead's process will not clear the failure; the next session of meaningful context size will reproduce it. Harness-level fixes (Solutions 1 and 3 in the Follow-up Actions table) are the only path to eliminating the failure mode.

### Stage 6a: File-writer attribution â€” team-lead's own process is the verified writer

Receiver-side (Stages 1â€“5) and sender-side process-instance analysis (Stage 6) both operated under an implicit assumption: that `role: assistant` entries in `team-lead`'s session jsonl reflect `team-lead`'s own process's output. lucas42 raised this as an unverified premise â€” if SendMessage works by direct file write, another agent's process could in principle have written into team-lead's jsonl, including a phantom-shaped `role: assistant` entry. `lucos-security` investigated.

**Verdict: team-lead's own process is the verified writer. No cross-process write is consistent with the evidence.** Four signals, in increasing order of decisiveness:

1. **File permissions.** All session jsonls are `rw-------` (mode 600), owned by `lucas`; the directory is `drwx------` (700). Since every claude.exe process runs as `lucas`, permissions don't structurally isolate them. Cross-write is permitted; this signal alone can't settle the question.
2. **lsof sweep.** While both `team-lead` (PID 1995621) and `lucos-site-reliability` (PID 1997800) were active, neither had any session jsonl file open. The SRE agent's only open file was the `claude.exe` binary itself. No process held another agent's file open for persistent writing. Cross-writes via open-write-close are still possible in principle, but no persistent channel was found.
3. **inotify watches (the key signal).** `team-lead`'s process held 5 inotify watches: `~/.claude/tasks/lucos-all-hands`, `~/.claude/settings.json`, `~/sandboxes/.claude/settings.local.json`, `~/sandboxes/.claude/`, `~/.claude/`. **It did not watch `~/.claude/projects/` (the session files directory).** SendMessage delivery to team-lead therefore does *not* go through a session-file watch â€” it goes through the harness's own IPC channel (stdin/stdout, all on `/dev/pts/1`). Under this model, each process writes `role: assistant` entries for its own LLM output and `role: user` entries for harness-delivered inputs (inbound SendMessages, tool results); no other process writes `role: assistant` content into another process's session file. A phantom-shaped `role: assistant` entry cannot be produced by an external process without impersonating the harness's stdin pipe to that process.
4. **Timing + content cross-check (the smoking gun).** Phantom line 677 in `73e07f83` was timestamped `2026-05-13T22:27:24Z`. At that moment in `lucos-site-reliability`'s session (`08b44d67`), the SRE was at the following actual activity: 22:27:12Z receiving commit result `89111e5` ("Replace TBD links in Follow-up Actions with filed issue numbers"); 22:27:31Zâ€“22:27:39Z responding and converting PR #147 to *draft*. The phantom claims: *"Both changes in commit `e7a8b21` on the report branch."* The real commit at that moment was `89111e5` and the SRE was converting the PR to draft â€” the opposite of "addressed, awaiting re-review." A writer with access to the SRE's actual session file would have known the real commit hash. The mismatch between the phantom's content and the receiver's actual activity is the strongest evidence that the phantom is hallucinated by team-lead's own process, not copied or inferred from another agent's state.

The one honest caveat: because the Claude Code binary is compiled and `lucos-security` could not read the IPC source code directly, no line-number citation is available for "each process writes only to its own jsonl." What is available is the behavioural evidence above â€” all four signals consistent with the own-process-only-writer model, and the timing+content mismatch inconsistent with any other interpretation.

**Implications.** The Stage 1 root cause (model emits phantom `<teammate-message>` blocks inside its own assistant output; harness re-feeds them as if they were inbound) stands. Solutions 1 and 3 remain the right fix. No structural revision to the report is needed.
